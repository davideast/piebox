# Explanation (Understanding-Oriented)

Explanation is for the user who wants to understand the "why". It zooms out to look at the system from a theoretical perspective.

## Key Principles & The "Reference vs. Explanation" Tension

- **Broaden the context:** Explain the design philosophy, historical decisions, and alternatives considered.
- **Weave things together:** Show how Module A interacts with Module B.
- **Visuals:** Use architecture diagrams and flowcharts.

## Writing Rules

- **Tone:** Discursive, narrative, authoritative.
- **Format:** Essays, articles, and logical arguments.
- **Anti-patterns:** Do not provide step-by-step instructions. Do not list exhaustive API endpoints. If you are telling the user to type a command, you have left Explanation.
