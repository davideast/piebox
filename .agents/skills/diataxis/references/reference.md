# Reference (Information-Oriented)

Reference material is for consultation. It describes the machinery. It is completely devoid of instruction.

## Key Principles & The "Reference vs. Explanation" Tension

- **Austere and factual:** State the facts. Nothing more.
- **Structure dictates form:** The layout should map 1:1 to the code, API, or CLI architecture. Alphabetical or logical grouping is required.
- **Completeness:** Reference must be exhaustive. If a parameter exists, it must be documented, even if it is rarely used.

## Writing Rules

- **No narrative:** Never use "We" or "You".
- **Format:** Rely heavily on tables, bullet points, and syntax blocks.
- **Examples:** Provide minimal code snippets showing _usage_, but do not explain _how to build_ the usage.
- **Anti-patterns:** Do not say "To create a user, do X" (that's a how-to). Say: "The `/create` endpoint accepts the following parameters."
