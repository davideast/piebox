## Finding: piebox today depends on *two* independent agent SDKs

The server-side path (`src/`) and the browser playground
(`examples/browser/src/`) consume **different agent SDKs**:

| Surface                   | Agent SDK consumed                  |
| ------------------------- | ----------------------------------- |
| `src/` (piebox core)    | `@earendil-works/pi-coding-agent` |
| `examples/browser/src/` | `@inbrowser/agent`                |

Neither shares types with the other. The portability and decoupling
work has to deal with both surfaces, not just one. Replacing one SDK
with the other would not improve the architecture; the right move is
a piebox-native Layer 2 that both SDKs can adapt to.

## Detailed leak: `@earendil-works/pi-coding-agent` into `src/operations/`

Even the layer that *should* be the most protocol-neutral â€” pure
filesystem operations â€” pulls its types from an external agent SDK:

```
import type { BashOperations } from "@earendil-works/pi-coding-agent";
import type { EditOperations } from "@earendil-works/pi-coding-agent";
import type { FindOperations } from "@earendil-works/pi-coding-agent";
import type { GrepOperations } from "@earendil-works/pi-coding-agent";
import type { LsOperations } from "@earendil-works/pi-coding-agent";
import type { ReadOperations } from "@earendil-works/pi-coding-agent";
import type { WriteOperations } from "@earendil-works/pi-coding-agent";
```

Each of these operation type aliases (`ReadOperations`,
`WriteOperations`, etc.) defines the operation's signature inside
`@earendil-works/pi-coding-agent`. piebox's `src/operations/*`
implements them but doesn't own them.

**Implication for the refactor:** Layer 2 cannot simply re-export
these types â€” it has to define them piebox-native and have the
existing operations satisfy *those* signatures structurally. Then
the agent SDK can adapt in the driver layer.

Of these, the **load-bearing types** for the agent driver (would have
to be re-shaped or adapted in a refactor):

- `ToolHandler` â€” the per-tool interface every `buildTools` entry implements
- `ToolContext` â€” the runtime arg passed to `execute()` (carries `workspace`, `runtime`, `signal`)
- `ToolResult` â€” the return shape (`ok`, `summary`, `data`)
- `SessionEvent` â€” the union the UI `switch`es over
- `ChatMessage` â€” used both for store data AND for SDK history round-trip
- `LlmClient` / `LlmConfig` â€” the model provider seam
- `Tracer` â€” optional, but agent-SDK-shaped

The **structural types** (`JsonSchema`, `ChatEvent`) are commodity
shapes that piebox-native equivalents could subsume.

## Implications for the refactor (rough â€” to be revised against C/G)

1. **Layer 2 must define its own operation types.** `src/operations/`
   currently can't be lifted into a protocol-neutral surface because
   its types come from `@earendil-works/pi-coding-agent`. The refactor
   has to introduce piebox-native `ReadOperations` /
   `WriteOperations` / etc. type aliases â€” then both SDKs adapt.

2. **The agent driver wraps two SDKs, not one.** A `@piebox/driver-agent`
   package would have to either (a) pick one SDK and have the other
   adapt to it, (b) ship sub-drivers for both, or (c) define its own
   agent-driver-internal types and have both SDKs adapt. Option (c)
   is consistent with the Layer 2 principle.

3. **`ToolContext` carries fields the tools don't use.** The
   browser playground's tools never read `ctx.workspace` or
   `ctx.runtime` â€” only `ctx.signal`. The piebox-native tool
   descriptor can be much leaner: `(args, sandbox, signal) =>
   PieboxResult<T>`.

4. **`SessionEvent` has dead branches in the playground.** The
   playground does not consume `strategy_event`, `workspace_changed`,
   or `runtime_changed`. If MCP and CLI drivers also don't, those
   shapes are pure agent-driver concern and never need to exist in
   Layer 2.

5. **Scope of the refactor (rough):** 16 `src/` files touch the
   server-side SDK; 2 `examples/browser/src/` files touch the
   browser-side SDK. The browser surface is smaller and likely
   cheaper to refactor first.
